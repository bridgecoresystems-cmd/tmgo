# Правильная архитектура backend — универсальное ТЗ

Стек: Elysia.js + Bun + Drizzle ORM + Better Auth  
Применимо к: langschool, ERP, любой новый проект

---

## Структура папок

```
src/
  modules/          ← основная бизнес-логика, разбитая по фичам
    schedule/
      index.ts      ← регистрирует модуль, подключает guard + routes
      routes.ts     ← только приём запроса и отдача ответа
      service.ts    ← вся бизнес-логика
      guard.ts      ← кто имеет доступ
      schema.ts     ← валидация входящих данных (Elysia t.Object)
    attendance/
      ...
    chat/
      ...
    billing/
      ...

  db/               ← всё про базу данных
    index.ts        ← подключение Drizzle
    schema/         ← таблицы (одна таблица = один файл)
      users.ts
      lessons.ts
      payments.ts

  models/           ← общие TypeScript типы которые используют все модули
    user.ts
    lesson.ts
    payment.ts

  auth/             ← Better Auth, сессии, токены
    index.ts

  constants/        ← константы проекта
    roles.ts        ← список всех ролей
    statuses.ts     ← статусы уроков, платежей и тд

  lib/              ← вспомогательные утилиты
    errors.ts       ← стандартные ошибки (NotFound, Forbidden и тд)
    response.ts     ← стандартный формат ответа API

  index.ts          ← точка входа, собирает все модули
```

---

## Принцип каждого слоя

### routes.ts — официант
Принимает запрос, вызывает service, возвращает ответ.  
Никакой логики. Никаких запросов к базе. Только передача данных.

```
запрос → валидация → service → ответ
```

### service.ts — повар
Вся бизнес-логика здесь. Записать студента на урок, проверить что место есть,
отправить уведомление учителю — всё это service.  
Не знает ничего про HTTP. Не знает про запросы и ответы.

### guard.ts — охранник
Проверяет сессию и роль. Если нет доступа — останавливает запрос до того
как он дойдёт до routes.ts.

### schema.ts — паспортный контроль
Описывает как должны выглядеть входящие данные.  
Elysia автоматически вернёт 422 если данные не совпадают со схемой.

### db/schema/ — чертёж базы данных
Только описание таблиц через Drizzle. Никакой логики.

### models/ — общий язык проекта
TypeScript типы которые используются везде.  
Если тип нужен только внутри одного модуля — он живёт внутри модуля.  
Если тип используют несколько модулей — он живёт в models/.

### constants/ — одно место правды
Список ролей, статусов, лимитов. Никогда не пиши магические строки
вроде `role === 'teacher'` прямо в коде — только через константы.

### lib/ — общий инструментарий
Стандартные ошибки и стандартный формат ответа.  
Все модули используют одно и то же — клиент получает предсказуемую структуру.

## Правила которые нельзя нарушать

1. **Модуль не импортирует другой модуль**  
   `schedule` не импортирует ничего из `attendance`.  
   Если нужны общие данные — они идут через `models/` или `db/`.

2. **Никакой логики в routes.ts**  
   Если в routes.ts больше 5 строк на один эндпоинт — логика попала не туда.

3. **Никаких запросов к базе в routes.ts**  
   Только service.ts работает с db.

4. **Один файл = одна ответственность**  
   routes.ts не валидирует. schema.ts не делает запросы. service.ts не знает про HTTP.

5. **Все ошибки через lib/errors.ts**  
   Не пиши `set.status = 403; return { error: 'Forbidden' }` в каждом файле.  
   Один раз описал класс ошибки — везде бросаешь его.

6. **Все роли через constants/roles.ts**  
   Не пиши строки ролей напрямую в коде.

## Как выглядит запрос от начала до конца

```
HTTP запрос
    ↓
index.ts (точка входа)
    ↓
guard.ts (есть сессия? правильная роль?)
    ↓
schema.ts (правильные данные?)
    ↓
routes.ts (принять запрос)
    ↓
service.ts (выполнить логику)
    ↓
db/ (запрос к базе)
    ↓
routes.ts (отдать ответ)
    ↓
HTTP ответ
```

## Как добавить новую фичу

1. Создать папку `src/modules/новая-фича/`
2. Создать `guard.ts` — указать кому разрешено
3. Создать `schema.ts` — описать входящие данные
4. Создать `service.ts` — написать логику
5. Создать `routes.ts` — подключить guard, schema, вызвать service
6. Создать `index.ts` — собрать всё вместе
7. Подключить модуль в `src/index.ts` через `.use()`

Больше ничего не трогаешь. Остальной проект не знает что появилась новая фича.

---

## Как выглядит index.ts точки входа

```
app
  .use(authModule)
  .use(scheduleModule)
  .use(attendanceModule)
  .use(chatModule)
  .use(billingModule)
  .use(studentsModule)
  .use(notificationsModule)
```

Каждая строка — независимый модуль.  
Убрал строку — фича отключена. Добавил — фича включена.

## Чеклист перед тем как считать модуль готовым

- [ ] guard.ts проверяет сессию и роль
- [ ] schema.ts описывает все входящие данные
- [ ] service.ts содержит всю логику, не знает про HTTP
- [ ] routes.ts не содержит логики, только вызовы service
- [ ] модуль не импортирует другие модули
- [ ] ошибки через lib/errors.ts
- [ ] роли через constants/roles.ts
- [ ] типы данных через models/ (если используются в других модулях)
